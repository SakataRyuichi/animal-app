# `/review` コマンド

コード変更をレビューし、潜在的な問題を検出します。

## ワークフロー

1. **変更内容の確認**
   - `git diff`で変更内容を確認
   - ステージされた変更とステージされていない変更を区別

2. **コードレビュー**
   - 変更されたファイルを読み込む
   - 以下の観点でレビュー：
     - **型安全性**: TypeScriptの型エラーがないか
     - **コードスタイル**: プロジェクトのコーディング規約に従っているか
     - **関数型プログラミング**: 副作用を最小限にしているか
     - **変数名**: 読みやすい形式になっているか
     - **エラーハンドリング**: 適切にエラーを処理しているか
     - **パフォーマンス**: 不要な再レンダリングや計算がないか
     - **セキュリティ**: セキュリティ上の問題がないか

3. **リントの実行**
   - `pnpm lint`を実行
   - ESLintのエラーや警告を確認

4. **問題の報告**
   - 発見した問題をリスト化
   - 各問題に対して具体的な修正提案を提示

5. **修正の提案**
   - 重大な問題がある場合は、修正を提案
   - 軽微な問題は、改善提案として提示

## 使用例

```
/review
```

または、特定のファイルをレビュー：

```
/review @src/components/Button.tsx
```

## レビュー観点

### 必須チェック項目

- [ ] TypeScriptの型エラーがない
- [ ] ESLintのエラーがない
- [ ] 関数型プログラミングの原則に従っている
- [ ] 変数名が読みやすい形式になっている
- [ ] エラーハンドリングが適切

### 推奨チェック項目

- [ ] テストが追加されている（新機能の場合）
- [ ] ドキュメントコメントが追加されている（公開APIの場合）
- [ ] パフォーマンスの最適化が考慮されている
- [ ] アクセシビリティが考慮されている（UIコンポーネントの場合）

## 注意事項

- レビューは建設的で、具体的な改善提案を含める
- コードの意図を理解してからレビューする
- プロジェクトのコーディング規約に従う
